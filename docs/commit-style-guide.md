## 1. Общие сведения

### 1.1. Структура сообщения

`<type>(<scope>): <краткое описание>`

### 1.2. Правила оформления

1. Длина заголовка коммита не должна превышать 72 символов;
2. Текст коммита на русском языке;
3. Все буквы должны быть строчными;
4. Вместо буквы `ё` в тексте коммита используется буква `е`;

### 1.3. Тело сообщения

Если коммит требует пояснения, после заголовка оставляется пустая строка и пишется тело. Тело может содержать:

- перечень исправлений с пометкой `исправлено:`;
- перечень изменений с пометкой `изменено:`;
- контекст принятого решения, почему был выбран именно этот подход.

Каждый пункт списка нумеруется и начинается со строчной буквы:

```text
fix(ui): исправлен z-index анимированного фона

исправлено:

1. баг, из-за которого орбы перекрывали ui-элементы.

2. баг, из-за которого кнопки тренировки были не видны на главной.
```

## 2. Типы коммитов (type)

| Тип      | Назначение                                                              |
| -------- | ----------------------------------------------------------------------- |
| feat     | Новая функциональность (новая фича, компонент, хоткей)                  |
| fix      | Исправление бага, ошибки линтера, регрессии                             |
| refactor | Переработка кода без изменения внешнего поведения                       |
| docs     | Изменения в документации                                                |
| style    | Форматирование, пробелы, переименование — не влияет на логику           |
| chore    | Рутинные задачи: обновление зависимостей, настройка сборки              |
| ci       | Изменения в конфигурациях CI/CD (GitHub Actions, линтеры)               |
| perf     | Изменения, направленные на повышение производительности                 |
| test     | Добавление или обновление тестов                                        |

## 3. Области (scope)

Области соответствуют основным компонентам yappari. При необходимости можно добавлять новые, но перечисленные ниже — стандартные.

| Область   | Описание                                                                |
| --------- | ----------------------------------------------------------------------- |
| `ui`      | Фронтенд (Vue 3 + TypeScript): компоненты, стили, анимации, composables |
| `backend` | Go-бэкенд: бизнес-логика, IPC-методы, app.go, main.go                   |
| `db`      | База данных: схемы SQLite, миграции, CRUD-методы                        |
| `sm2`     | Алгоритм SM-2: логика интервального повторения                          |
| `tts`     | TTS-модуль (Silero + Kokoro): Go-обёртка + Python-скрипты               |
| `docs`    | Документация: README, руководства, commit-style-guide                   |
| `ci`      | CI/CD: GitHub Actions, линтеры, настройка сборки                        |
| `config`  | Конфигурация: .gitignore, wails.json, зависимости                       |

---

## 4. Примеры корректных сообщений

- `feat(ui): добавлен профиль-поповер с информацией о приложении`
- `feat(ui): полноэкранный режим по хоткею f`
- `fix(ui): z-index анимированного фона`
- `fix(lint): исправление ошибок ruff и golangci-lint`
- `docs(readme): финальная версия readme`
- `docs: докстринги для go и vue модулей`
- `docs: добавлен commit-style-guide для проекта`
- `refactor(tts): json-протокол с примерами в tts_server.py`
- `ci(github): добавлен python setup + ruff линтинг в build.yml`
- `perf(tts): web speech как основной tts`

---

### 4.1. Описание багов

При описании исправления багов используй формат:

```text
fix(scope): краткое описание

исправлено:

1. баг, из-за которого <описание>.

2. баг, из-за которого <описание>.
```
